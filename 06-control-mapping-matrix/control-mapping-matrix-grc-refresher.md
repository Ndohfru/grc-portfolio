# Control Mapping Matrix – GRC Refresher Notes

## What it is

A control mapping matrix is a structured document that shows how a single control theme aligns across multiple frameworks, standards, and regulations.

It helps answer:
- what the control theme is
- what business risk the control addresses
- which frameworks expect the control
- how the control overlaps across standards
- who typically owns the control
- what evidence could support it
- how mature the control currently appears

In GRC terms, a control mapping matrix is a harmonization artifact that connects:
- governance
- risk
- compliance
- control rationalization
- audit readiness
- framework comparison

## Why it matters

Most organizations do not operate against just one framework.

They may simultaneously face:
- PCI DSS
- NIST CSF
- ISO 27001
- CIS Controls
- GDPR / CCPA
- customer security expectations
- audit requirements

Without mapping, teams often treat these as separate obligations and create duplicated effort.

A control mapping matrix forces a more mature view:
- one control can support several obligations
- some frameworks are more prescriptive
- some are broader and outcome-based
- privacy relevance may be direct or indirect depending on the control

That is why control mapping matters in:
- compliance program design
- audit preparation
- control rationalization
- remediation planning
- framework harmonization
- GRC strategy

## The logic of the main columns

### Control ID
A unique identifier for each mapped control theme.

Why it matters:
- supports traceability
- makes discussion easier
- allows future expansion and maintenance

Professional explanation:
> The Control ID provides a consistent reference point for each control theme across the matrix.

### Control Theme
This is the main control category being analyzed.

Examples:
- access control
- MFA
- logging and monitoring
- vendor management
- incident response
- encryption
- governance oversight

Professional explanation:
> The Control Theme identifies the broad safeguard or process area being compared across frameworks.

### Control Objective
This explains what the control is supposed to achieve.

Examples:
- restrict access based on business need
- protect sensitive data through encryption
- detect incidents through logging and monitoring

Professional explanation:
> The Control Objective defines the intended outcome of the control and clarifies why the control exists.

### Business Risk Addressed
This connects the control to real organizational exposure.

Examples:
- unauthorized access
- data leakage
- service disruption
- compliance failure
- breach impact
- weak accountability

Professional explanation:
> This column links the control to the specific business, security, privacy, or compliance risk it is intended to reduce.

### PCI DSS Reference
This shows the relevant PCI DSS requirement tied to the control.

Why it matters:
PCI DSS is more prescriptive than many broader frameworks, especially in payment-related environments.

Professional explanation:
> The PCI DSS Reference identifies the payment-security requirement that aligns with the mapped control theme.

### NIST CSF Function
This maps the control to the relevant NIST Cybersecurity Framework function.

Examples:
- Identify
- Protect
- Detect
- Respond
- Recover
- Govern

Professional explanation:
> The NIST CSF Function shows where the control sits within the broader cybersecurity lifecycle.

### ISO 27001 Reference
This maps the control to an ISO 27001 clause or Annex A control area.

Why it matters:
ISO 27001 helps show how control themes fit into a governance and management-system context.

Professional explanation:
> The ISO 27001 Reference links the control theme to a recognized information security management and control domain.

### CIS Controls Reference
This maps the control to CIS Controls.

Why it matters:
CIS helps translate control themes into more practical and operational security action.

Professional explanation:
> The CIS Controls Reference links the control theme to a practical security-control baseline that is often more implementation-oriented.

### GDPR / CCPA Relevance
This shows whether the control has privacy relevance, and whether that relevance is direct or indirect.

Examples:
- directly relevant to privacy obligations
- indirectly relevant where personal data is involved
- highly relevant during breach handling

Professional explanation:
> This column captures how the control intersects with privacy obligations, especially where personal data access, protection, retention, or incident response is involved.

### Typical Owner
This identifies the internal function typically accountable for the control.

Examples:
- IAM Manager
- Security Operations Lead
- Data Protection Officer
- Vendor Management Lead
- Head of Compliance

Professional explanation:
> The Typical Owner reflects the function that would usually be accountable for operating, monitoring, or coordinating the control.

### Example Evidence
This lists the kinds of artifacts that could demonstrate the control exists and is operating.

Examples:
- access review reports
- scan results
- incident response plans
- policies
- retention schedules
- due diligence checklists

Professional explanation:
> Example Evidence shows what artifacts could be used to validate the control during review, assessment, or audit.

### Control Maturity / Status
This gives a simple view of how developed or effective the control appears.

Examples:
- Partially Implemented
- Needs Improvement
- Implemented
- Under Review

Professional explanation:
> Control Maturity / Status provides a quick view of the current state of the control and whether it appears mature, partial, or weak.

### Notes / Mapping Insight
This is one of the most valuable columns.

It captures nuance such as:
- where one framework is more prescriptive
- where another is broader
- where privacy impact becomes direct
- where governance plays an umbrella role

Professional explanation:
> Notes / Mapping Insight captures the analytical value of the matrix by explaining overlap, differences, emphasis, and practical interpretation across frameworks.

## Helpful distinctions

### Control mapping vs gap assessment
- Control mapping shows alignment and overlap across frameworks
- Gap assessment evaluates whether the control is actually present, effective, or deficient in the environment

### Framework overlap vs framework equivalence
Just because two frameworks touch the same control theme does not mean they are identical.

Example:
- PCI DSS may be more prescriptive
- NIST CSF may be broader and functional
- ISO 27001 may emphasize governance structure
- GDPR / CCPA may only become relevant when personal data is involved

### Evidence vs mapping
Mapping shows that a control should exist.
Evidence shows whether it actually exists and is operating.
